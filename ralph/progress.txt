## Codebase Patterns
- `filter create` CLI uses `--file <FILE>` not inline `--name`/`--type`/`--config` flags — JSON spec file required
- `filter delete` prompts for confirmation (y/N) — pipe `echo "y"` for scripting
- OAuth2 is the only filter with `allowedAttachmentPoints: ["listener"]` — all others support route+listener
- OAuth2 auto-references an SDS secret named `hmac-secret` for cookie signing (hardcoded in `to_any()`)
- MCP filter `reject_no_mcp` mode requires custom `envoy.filters.http.mcp` extension in Envoy — standard dev image may not include it
- `header_mutation` does NOT require listener-level config (`requires_listener_config: false`) — works with empty config
- `custom_response` matcher uses tagged union for status codes: `{"type": "exact", "code": 503}` or `{"type": "range", "min": 500, "max": 599}`
- Compressor filter has a known Envoy NACK bug: `RuntimeFeatureFlag.RuntimeKey` is empty in `src/xds/filters/http/compressor.rs` `to_any()` (line ~266) — Envoy requires non-empty runtime_key
- `rate_limit` (external) is rejected by the API with 400 — only `local_rate_limit` works for rate limiting
- `router` is NOT a Flowplane FilterType — it's an internal Envoy filter added automatically to every listener
- CORS filter config wraps everything in a `policy` object: `{"config": {"type": "cors", "config": {"policy": {...}}}}`
- Compressor library uses tagged enum: `{"type": "gzip", "compression_level": "default_compression"}` — not nested like other filter configs
- `import` and `learn` are CLI commands in the release/v0.2.0 binary but NOT in the current branch source under `src/cli/` — use `git show release/v0.2.0:src/cli/import.rs` to read source
- `import openapi` auto-resolves the dataplane ID by listing dataplanes — no `--dataplane-id` flag needed
- `learn start` default output is JSON; `learn list` default output is table — asymmetric defaults
- Learning session response uses camelCase (`routePattern`, `targetSampleCount`, `currentSampleCount`, `progressPercentage`)
- `docs/quickstart.md` was already deleted before this PRD started — always verify file existence before planning deletion work
- The `expose` command source is on `release/v0.2.0` branch at `src/cli/expose.rs`, not on `main` — Docker stack must be built from release/v0.2.0 for the expose API endpoint to work
- The installed `flowplane` binary at `~/.local/bin/flowplane` may have more commands than the current branch source — always check `flowplane --help` against the live binary
- To test docs against a live stack when current branch lacks server features: `git stash && git checkout release/v0.2.0 && flowplane init ... && git checkout <branch> && git stash pop`
- Filter JSON uses camelCase for top-level API fields (`filterType`, not `filter_type`) — the Rust struct has `#[serde(rename_all = "camelCase")]`
- Filter config is nested: `"config": {"type": "local_rate_limit", "config": {...actual settings...}}` — custom Serialize/Deserialize on FilterConfig enum
- `local_rate_limit` is the implemented in-memory rate limiter; `rate_limit` (external/distributed) exists as FilterType but has `is_implemented: false`
- Listener naming convention from `expose`: `<name>-listener` — needed for `filter attach --listener`
- MCP endpoints: `/api/v1/mcp/cp` (control plane) and `/api/v1/mcp/api` (gateway API) — both Streamable HTTP
- `flowplane down` preserves database volume; `flowplane down --volumes` deletes it
- Prod mode uses `make up ENVOY=1 HTTPBIN=1` + `make seed` + `flowplane auth login` — requires Zitadel
- Prod mode xDS port is 50051 (vs 18000 in dev mode)
- `flowplane list` strips the `-listener` suffix for display — `demo-listener` shows as `demo`
- `flowplane status <name>` requires the full listener name (e.g., `demo-listener`), not the short display name
- `flowplane doctor` checks CP health and Envoy readiness at localhost:9901 — skips Envoy check for remote servers
- `flowplane database` subcommands require `--database-url` or `DATABASE_URL` env var — `init` handles migrations automatically
- Listener create requires `dataplaneId` — dev mode uses `dev-dataplane-id`; expose handles this automatically
- Filter delete returns 409 if filter is still attached — must detach first
- Cluster update JSON must include the `name` field (400 error otherwise)
- Listener filter chain type is `httpConnectionManager` (camelCase), NOT `HttpConnectionManager` (PascalCase)
- Rate limit denominator values are snake_case: `hundred`, `ten_thousand`, `million` — NOT SCREAMING_CASE
- Dev mode token is stored at `~/.flowplane/credentials` (plain text), not in config.toml JWT
- Dev mode token env var: `FLOWPLANE_DEV_TOKEN` on the control plane container
- Secret create `--config` JSON must include `"type"` field matching `--type` flag — the live binary does NOT auto-inject it (source code has inject logic but may not be in the release binary)
- Secret config uses `configuration` field in the API request body (camelCase: `secretType`, `expiresAt`, `createdAt`, `updatedAt`)
- Valid secret types: `generic_secret`, `tls_certificate`, `certificate_validation_context`, `session_ticket_keys`
- Token resolution priority: `--token` > `--token-file` > `~/.flowplane/credentials` > `~/.flowplane/config.toml` > `FLOWPLANE_TOKEN` env var
- `auth whoami` in dev mode shows "opaque (dev mode token)"; in prod mode decodes JWT and fetches userinfo from Zitadel
- `config show` default output is YAML; `config set` key validation is compile-time via clap value_parser
- `config.toml` keys include `oidc_issuer`, `oidc_client_id`, `callback_url` (not documented in user-facing help but accepted)
- MCP filter's Envoy name is `envoy.filters.http.mcp`, NOT `envoy.filters.http.ext_proc` — the old docs/filters.md was wrong
- Per-route `typedPerFilterConfig` keys use Envoy HTTP filter names (e.g., `envoy.filters.http.local_ratelimit`), not Flowplane type names
- The old filters.md listed phantom types (rate_limit_quota, health_check, credential_injector, ext_proc, router) not in `src/domain/filter.rs` — always verify against the FilterType enum
- JWT auth `local` JWKS must contain at least one valid public key — empty `{"keys":[]}` causes Envoy NACK: "Jwks doesn't have any valid public key"
- JWT auth `local` JWKS inline_string must have properly encoded RSA `n` and `e` fields — malformed base64url causes NACK: "Jwks RSA [n] or [e] field is missing or has a parse error"
- RouteMatchConfig path uses PascalCase enum variant names: `{"Prefix": "/"}`, `{"Exact": "/path"}`, `{"Regex": ".*"}` — NOT snake_case
- ext_authz gRPC `target_uri` is an Envoy cluster name (not a URL) — the cluster must exist or requests will fail
- ext_authz with `failure_mode_allow: false` returns 403 when the authz service is unreachable
- RBAC filter requires at least `rules` or `shadow_rules` — empty config fails validation
- RBAC permission/principal types use tagged union with `"type"` field: `{"type": "header", "name": ":method", "exact_match": "GET"}`
- `credential_injector` is NOT a FilterType — it exists as an XDS module (`src/xds/filters/http/credential_injector.rs`) but is not registered in `src/domain/filter.rs` FilterType enum
- Available filter types (from API error): header_mutation, jwt_auth, local_rate_limit, custom_response, mcp, cors, compressor, ext_authz, rbac, oauth2 (10 implemented + rate_limit defined but not implemented = 11 in enum)
