# Spec 02 ŌĆö MCP Server & Tool Catalog

Behavioral specification extracted from Flowplane v1 (verified against v0.2.10 source at
`/tmp/flowplane-v1`). This document is self-contained: it specifies the MCP server, the
complete static toolset (82 tools), the dynamic `api_*` gateway-tool family, and the
resources/prompts surfaces, sufficient to reimplement without reading v1 source.

v1 source anchors are cited per-section as `[v1: path]`.

## 1. MCP Server Overview

`[v1: src/mcp/streamable_http/, src/mcp/handler.rs, src/mcp/protocol.rs, src/api/routes.rs]`

### 1.1 Endpoint & transport

- **Single unified endpoint**: `/api/v1/mcp` on the main API server (Axum), serving both
  control-plane (`cp_*`/`ops_*`/`devops_*`/`dev_*`) and gateway (`api_*`) tools from one session.
  - `POST /api/v1/mcp` ŌĆö JSON-RPC 2.0 request (body = `JsonRpcRequest`). Response is JSON
    (default) or routed to an attached SSE stream (202 Accepted) when `Accept` contains
    `text/event-stream` AND the session has an SSE connection attached.
  - `GET /api/v1/mcp` ŌĆö opens an SSE stream for serverŌåÆclient notifications (requires existing
    session). Supports resumption via `Last-Event-ID`.
  - `DELETE /api/v1/mcp` ŌĆö terminates the session (200 OK; 404 if unknown; 400 if header
    missing/malformed).
- **Ancillary REST**: `GET /api/v1/mcp/connections` lists active SSE connections + HTTP
  sessions for a team (authz: `mcp:read` on the team; admins must pass `?team=`; team is
  validated to belong to caller's org). `GET /api/v1/admin/mcp/connections/summary` is an
  admin summary endpoint. `[v1: src/mcp/connections_api.rs]`
- Transport is **Streamable HTTP per MCP spec 2025-11-25**. There is no stdio transport in v1
  (a former stdio server was removed; `src/mcp/mod.rs` carries the comment).

### 1.2 Protocol version

`[v1: src/mcp/protocol.rs:13-19]`

- Preferred/advertised version: **`2025-11-25`** (`PROTOCOL_VERSION`).
- Accepted versions: **`["2025-11-25", "2025-03-26"]`** (`SUPPORTED_VERSIONS`). Validation is
  exact-match against this list, applied to (a) the `protocolVersion` field of `initialize`
  params and (b) the optional `MCP-Protocol-Version` HTTP header when present. Unsupported
  versions ŌåÆ JSON-RPC error `-32600` with `data.supportedVersions` listing the accepted set.
- The server echoes the client's requested version back in `InitializeResult.protocolVersion`
  (it does not force the preferred version).

### 1.3 HTTP headers

`[v1: src/mcp/transport_common.rs, src/mcp/streamable_http/post_handler.rs]`

| Header | Direction | Semantics |
|---|---|---|
| `MCP-Protocol-Version` | req/resp | Optional on requests; validated if present. Echoed on every response (negotiated value, falling back to `2025-11-25`). |
| `MCP-Session-Id` | req/resp | Required on all non-`initialize` POSTs, all GETs, all DELETEs. Generated server-side on `initialize` and returned on the response. Legacy fallback: `?sessionId=` query param on POST. |
| `Accept` | req | `text/event-stream` selects SSE response mode (only effective if session has an SSE connection); otherwise JSON. |
| `Last-Event-ID` | req (GET) | SSE resumption cursor, format `{connection_id}:{sequence}`. |
| `Origin` | req | Parsed/validated by `validate_origin_header` against `FLOWPLANE_MCP_ALLOWED_ORIGINS` (comma-separated `proto://host`, port-insensitive; defaults `http://localhost`, `http://127.0.0.1`, `http://[::1]`). **NOTE: in v1 this helper exists in `src/mcp/security.rs` but is NOT invoked by the streamable-HTTP handlers** ŌĆö see ┬¦7 gaps. |

### 1.4 Session model

`[v1: src/mcp/session.rs, src/mcp/security.rs]`

- Session IDs: `mcp-{uuid-v4}` (122 bits randomness), generated by
  `generate_secure_session_id()` on `initialize`; format strictly validated by regex
  `^mcp-[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$`
  (`validate_session_id_format`).
- `SessionManager` is an in-memory `DashMap<SessionId, McpSession>`; key is namespaced
  (`session:{id}` for header-derived IDs, `token:{id}` variant exists for token-derived IDs).
  **Default TTL 3600 s** since last activity; expired sessions are removed by
  `cleanup_expired()`. No persistence ŌĆö sessions die on restart.
- `McpSession` state: `initialized: bool`, `protocol_version`, `client_info`,
  `log_level` (default `info`), `teams: Vec<String>` (bound at creation from the token's
  grants), `created_at`/`last_activity`, `connection_id: Option<String>` (attached SSE stream).
- POST flow: `initialize` ŌåÆ new session created bound to the token's authorized teams
  (extracted via `extract_teams()` from grant team names; dev fallback `?team=` query param
  when token has org-admin but no grants); on success the session is marked initialized with
  negotiated version + clientInfo. Non-initialize POSTs require a valid, existing session ID.
  JSON-RPC notifications (requests with no `id`) return **202 Accepted, empty body**.
- **Security caveat (preserve or fix in v2)**: for existing sessions "the session ID is the
  security boundary ŌĆö any valid token that presents a known session ID may use it"
  (`post_handler.rs:186-191`). The handler is reconstructed per request from the *caller's*
  AuthContext, so authorization still derives from the presented token, but session/team
  binding is not re-validated against the token.

### 1.5 Initialization handshake

`[v1: src/mcp/handler.rs:273-344]`

`initialize` response:

- `serverInfo`: `{ name: "flowplane-mcp", version: <crate version>, title: "Flowplane MCP
  Server", description: "Envoy control plane management via Model Context Protocol" }`.
- `capabilities`: `tools: { listChanged: false }`, `resources: { subscribe: false,
  listChanged: false }`, `prompts: { listChanged: false }`, `logging: {}`. No completions,
  tasks, or experimental capabilities.
- `instructions`: a ~40-line constant (`SERVER_INSTRUCTIONS`) covering, verbatim topics:
  1. Resource creation order: clusters ŌåÆ route configs ŌåÆ filters (create then attach) ŌåÆ
     listeners (bind to route configs).
  2. Pre-creation checks: `cp_query_port`, `cp_query_path`, `dev_preflight_check`.
  3. xDS delivery semantics: NACK keeps previous config; one bad resource NACKs the whole
     batch of that type; use `ops_xds_delivery_status`.
  4. Route matching is first-match-wins, top-to-bottom; specific paths before broad prefixes.
  5. `cp_update_route_config` is a FULL REPLACEMENT of the `virtualHosts` array ŌĆö fetch with
     `cp_get_route_config` first and include all existing routes.
  6. Diagnostic workflow: `ops_topology` ŌåÆ `ops_trace_request` ŌåÆ `ops_config_validate` ŌåÆ
     `ops_xds_delivery_status` ŌåÆ `ops_nack_history`.
  7. Error recovery: ALREADY_EXISTS ŌåÆ query/reuse/rename; NOT_FOUND ŌåÆ create prerequisite;
     CONFLICT ŌåÆ inspect and resolve.

Handled JSON-RPC methods: `initialize`, `initialized`, `ping`, `tools/list`, `tools/call`,
`resources/list`, `resources/read`, `prompts/list`, `prompts/get`, `logging/setLevel`,
`notifications/initialized`, `notifications/cancelled` (both notifications are simply
acknowledged with `{}`). Anything else ŌåÆ `-32601 Method not found`. Initialization is NOT
enforced before other methods (stateless-HTTP concession; bearer auth is the boundary).

### 1.6 Authentication & per-call authorization

`[v1: src/mcp/handler.rs (check_tool_authorization, handle_tools_call),
 src/mcp/tool_registry.rs (get_tool_authorization), src/auth/authorization.rs
 (check_resource_access)]`

- **Authentication**: `/api/v1/mcp` sits behind the same bearer-token middleware
  (`auth::middleware::authenticate`, optionally Zitadel-backed) as the REST API. The middleware
  injects an `AuthContext` extension carrying: token id/name, scopes (e.g. `admin:all`,
  `org:{name}:{role}`), `grants` (rows with `grant_type` Ōłł {Resource, GatewayTool},
  `team_id`/`team_name`, `resource_type`, `action`, `route_id`, `allowed_methods`),
  `org_id`/`org_name`, `user_id`, and optional `agent_context` Ōłł {CpTool, GatewayTool,
  ApiConsumer}.
- **Per-call authorization pipeline for `tools/call`** (in order; each step fails closed):
  1. `api_*` names short-circuit to the gateway executor path (┬¦4.4).
  2. `get_tool_authorization(tool_name)` (static `LazyLock<HashMap>` in `tool_registry.rs`)
     maps the tool to `{resource, action, risk_level}`. Unknown non-`api_*` names ŌåÆ
     gateway-tool lookup if executor configured, else `ToolNotFound`.
  3. Team resolution: `args.team` (string, non-empty) else session's first team else `""`.
     If non-empty and the session has a team list, the team must be in `self.teams`
     (else `Forbidden: Team '{t}' is not accessible with your token scopes`).
  4. **`check_tool_authorization(tool_name, team)`** ŌåÆ calls
     **`check_resource_access(&AuthContext, resource, action, Some(team))`** ŌĆö the same
     function REST handlers use. Denial message:
     `Access denied: {tool} requires team:{team}:{resource}:{action}`.
  5. Team **nameŌåÆUUID** resolution (`resolve_team_uuid`) scoped by `org_id`
     (`SELECT id FROM teams WHERE name=$1 AND org_id=$2`); `org_id = None` with a team *name*
     is rejected (`Forbidden`) so `admin:all` governance sessions cannot smuggle cross-org
     team names. Empty team or already-a-UUID passes through.
- **`check_resource_access` semantics** (must be reproduced exactly):
  - Agent guard: `GatewayTool` and `ApiConsumer` agents are structurally denied ALL CP
    resources (return false). `CpTool` agents use Resource-grant matching only.
  - `admin:all` bypass applies **only to governance resources**:
    `admin, admin-orgs, admin-users, admin-audit, admin-summary, admin-teams, admin-scopes,
    admin-apps, admin-filter-schemas, organizations, users, teams, stats`
    (`is_governance_resource`). It does NOT grant tenant resources (clusters, routes, ...).
  - Otherwise: exact grant match `(resource, action, team_name)`; org admins
    (`org:{org}:admin` scope matching the context's own `org_name`) get implicit access to all
    teams in their org.
- **`tools/list` filtering** mirrors authorization (`handler.rs:357-562`):
  - `CpTool` agents: only CP tools whose `(resource, action)` is in their Resource grants.
  - `GatewayTool` agents: NO CP tools; only enabled `api_*` tools whose `route_id` is in
    their GatewayTool grants.
  - `ApiConsumer` agents: no tools at all.
  - Humans: CP tools filtered by Resource grants; if no grants, org admins (or `admin:all`
    without org) see ALL CP tools, others see none. Gateway tools listed per team only when
    the caller has `api:read` on that team; only `enabled` tools.
  - Every listed tool description is prefixed with **`[Risk: SAFE|LOW|MEDIUM|HIGH]`** from the
    registry. Gateway tools without an object `inputSchema` get
    `{"type":"object","properties":{},"additionalProperties":false}` substituted.
- **Risk levels** (`RiskLevel`, ordered): `SAFE < LOW < MEDIUM < HIGH < CRITICAL`. v1 only
  annotates; it does not yet gate on risk (comment: future `if risk >= High { require_approval }`).

### 1.7 JSON-RPC error mapping

`[v1: src/mcp/error.rs]`

`McpError` variants ŌåÆ codes: `ParseError`ŌåÆ-32700; `InvalidRequest`/`NotInitialized`/
`ToolNotFound`/`UnsupportedProtocolVersion`/`Forbidden`/`Conflict`/`InvalidOrigin`/
`MissingOrigin`/`MalformedSessionId`/`InvalidEventId`ŌåÆ-32600; `MethodNotFound`/
`ResourceNotFound`/`PromptNotFound`ŌåÆ-32601; `InvalidParams`/`ValidationError`ŌåÆ-32602;
`InternalError`/`GatewayError`/`DatabaseError`/`SerializationError`/`IoError`/
`ConnectionLimitExceeded`/`Configuration`ŌåÆ-32603. `UnsupportedProtocolVersion` adds
`error.data.supportedVersions`. Note: there is no dedicated 403 in JSON-RPC, so `Forbidden`
surfaces as -32600 with message prefix `Forbidden: `. Contract (fp-xsli.3.2): `Conflict`
(-32600) must never share a code with `InvalidParams`/`ValidationError` (-32602) because
clients branch on `error.code` for retry decisions.

Additionally, tool *business* errors are often returned as successful `tools/call` results
with `isError: true` and a structured text payload (see ┬¦3.0 output conventions).

### 2.1 Dispatch architecture (handler.rs)

Two execution paths for static tools:

1. **Direct-DB tools** (take `&DbPool` + team **UUID** + `Option<&OrgId>` + args):
   `cp_list_routes`, `cp_query_port`, `cp_query_path`, `cp_query_service`,
   `ops_trace_request`, `ops_topology`, `ops_config_validate`, `ops_audit_query`,
   `ops_xds_delivery_status`, `ops_nack_history`, `dev_preflight_check`.
2. **Internal-API tools** (take `&Arc<XdsState>` + team **NAME** + `Option<&OrgId>` + args;
   create/list ops additionally take `user_id` for audit): everything else. These reuse the
   same internal API layer as REST handlers, so validation, audit, and xDS push semantics are
   identical to REST. If `xds_state` is not configured the call fails with
   `InternalError("Operation not available: xds_state not configured")`.
   - Exception: the 10 team-scoped `cp_*_rate_limit_*` tools receive the team **UUID** plus
     the full `AuthContext`; the 2 `*_rate_limit_overrides` tools receive only `AuthContext`
     (governance, no team).

Tool executors share two helpers `[v1: src/mcp/tools/mod.rs:193-239]`:
- `validate_team_in_org(db, team_uuid, org_id)` ŌåÆ `Forbidden("Team '{t}' not found in your
  organization")` when the team isn't in the caller's org.
- `resolve_mcp_auth(team, org_id, team_repo)`: empty `team` is only legal for governance
  sessions with **no** org context (yields admin internal context); empty team + org context
  ŌåÆ `Forbidden("team-scoped MCP tools require an explicit `team` argument")` (fail-closed,
  fp-nw8b.13).

## 3. Complete Static Tool Catalog

Schemas below are exact (from the auto-generated reference cross-checked against the tool
modules). Format per tool:

> **name** ŌĆö description. *Auth:* `resource:action` (RISK). *Params:* `name:type` (req|opt,
> default) ŌĆö notes. *Out:* shape.

### 3.0 Output & error conventions (all modules)

`[v1: src/mcp/response_builders.rs, src/mcp/protocol.rs]`

- Every tool returns MCP `ToolCallResult { content: [ContentBlock::Text{text}], isError? }`.
  The `text` is serialized JSON (`serde_json::to_string_pretty` or compact). Content blocks
  support `text` and `image` types; tools only emit `text`.
- Standard token-efficient payloads (builders; target budgets in parens):
  - Query hit: `{"found": true, "ref": {"type","name","id"}, "data": {...}}` (50ŌĆō80 tok);
    miss: `{"found": false}`.
  - Action success: `{"ok": true, "ref": {...}}`; rich create adds `"details"`, `"created"`
    (child counts), `"next_step"` (workflow hint). Rich delete:
    `{"ok": true, "deleted": {"type","name"}, "cascade"?: {...}}`. Plain delete: `{"ok": true}`.
  - List: `{"items": [{"type","name","id"}...], "count": N}` (plus module-specific arrays,
    e.g. `"clusters": [...]`, `"sessions": [...]`).
  - Get: `{"ok": true, "ref": {...}, "data": {...}}`.
  - Structured business error (returned with `isError: true`):
    `{"ok": false, "error": "<CODE>", "existing"?: {...}, "fix": "<actionable hint>"}` with
    codes `ALREADY_EXISTS | NOT_FOUND | INVALID_CONFIG | DEPENDENCY | CONFLICT |
    MISSING_PARAM | INVALID_PARAM`.
  - Hard failures (authz, malformed args, DB errors) surface as JSON-RPC errors per ┬¦1.7.
- `ResourceRef.type` values: `listener, route, cluster, route_config, filter, virtual_host,
  dataplane, learning_session, openapi_import, aggregated_schema, filter_attachment`.
- All list tools accept pagination `limit:int` (opt, default 50, min 1, max 1000) and
  `offset:int` (opt, default 0, min 0) via shared `pagination_schema()`.
- All mutations are team-scoped via the resolved session/args team and recorded to the audit
  log; xDS-affecting mutations trigger a config push to connected Envoy dataplanes.

### 3.1 Clusters `[v1: src/mcp/tools/clusters.rs]`

- **cp_list_clusters** ŌĆö List upstream clusters. *Auth:* `clusters:read` (SAFE).
  *Params:* `limit`, `offset`. *Out:* list of cluster refs + `count`.
- **cp_get_cluster** ŌĆö Get cluster detail by name. *Auth:* `clusters:read` (SAFE).
  *Params:* `name:string` (req). *Out:* get-response with `service_name`, `configuration`
  (endpoints, lb_policy, timeouts, TLS, circuit breakers, health check), `version`,
  `created_at`/`updated_at`, `import_id`/`source` (tracks OpenAPI-import provenance).
- **cp_get_cluster_health** ŌĆö Aggregated endpoint health for a cluster. *Auth:*
  `clusters:read` (SAFE). *Params:* `name:string` (req). *Out:* per-endpoint
  `healthy|degraded|unhealthy|unknown` statuses, `total_endpoints`, summary `status`.
- **cp_create_cluster** ŌĆö Create upstream cluster. *Auth:* `clusters:create` (LOW). *Params:*
  - `name:string` (req) ŌĆö lowercase letters, digits, single hyphens; 1ŌĆō100 chars.
  - `serviceName:string` (req) ŌĆö descriptive service name.
  - `endpoints:array<object>` (req) ŌĆö `{host:string, port:int(1-65535)}` items.
  - `lbPolicy:string` (opt, `ROUND_ROBIN`) ŌĆö enum `ROUND_ROBIN|LEAST_REQUEST|RANDOM|RING_HASH|MAGLEV`.
  - `connectTimeoutSeconds:int` (opt, 5); `useTls:bool` (opt, false);
  - `healthCheck:object` (opt) ŌĆö `{path, intervalSeconds, timeoutSeconds, healthyThreshold, unhealthyThreshold}`;
  - `circuitBreakers:object` (opt) ŌĆö `{maxConnections, maxPendingRequests, maxRequests, maxRetries}`.
  *Out:* rich create response. *Errors:* `ALREADY_EXISTS` with `existing` + fix.
- **cp_update_cluster** ŌĆö Update cluster (partial; only provided fields change). *Auth:*
  `clusters:update` (MEDIUM). *Params:* `name` (req); `serviceName`, `endpoints`, `lbPolicy`,
  `connectTimeoutSeconds`, `useTls`, `healthCheck`, `circuitBreakers` (all opt). *Out:* update
  response. *Errors:* `NOT_FOUND`.
- **cp_delete_cluster** ŌĆö Delete cluster. *Auth:* `clusters:delete` (HIGH). *Params:*
  `name:string` (req). *Out:* delete response. *Errors:* `NOT_FOUND`; `DEPENDENCY` if routes
  still target it.
- **cp_query_service** ŌĆö Service-level summary for a cluster: endpoints, health, referencing
  route configs and listeners. *Auth:* `clusters:read` (SAFE). *Params:* `name:string` (req).
  *Out:* `{cluster, endpoints, route_configs, listeners, ...}` summary. (Direct-DB tool.)

### 3.2 Listeners `[v1: src/mcp/tools/listeners.rs]`

- **cp_list_listeners** ŌĆö List listeners. *Auth:* `listeners:read` (SAFE). *Params:* `limit`,
  `offset`. *Out:* list with `address`, `port`, `protocol` per item.
- **cp_get_listener** ŌĆö Get listener detail. *Auth:* `listeners:read` (SAFE). *Params:*
  `name:string` (req). *Out:* get-response incl. `configuration` (filter chains),
  `route_config`, `version`, provenance `import_id`/`source`.
- **cp_get_listener_status** ŌĆö Listener runtime status. *Auth:* `listeners:read` (SAFE).
  *Params:* `name:string` (req). *Out:* status incl. `active_connections`,
  `route_config_count`.
- **cp_query_port** ŌĆö Check if a port is already used by a listener. *Auth:* `listeners:read`
  (SAFE). *Params:* `port:int` (req, 1ŌĆō65535). *Out:* query response: `{"found":false}` or
  `{"found":true, "ref":{listener...}, "data":{address,port,...}}`. (Direct-DB tool.)
- **cp_create_listener** ŌĆö Create listener. *Auth:* `listeners:create` (MEDIUM). *Params:*
  - `name:string` (req); `port:int` (req, **>= 1024 ŌĆö privileged ports forbidden**);
  - `dataplaneId:string` (req) ŌĆö owning dataplane id; required so `api_*` tool execution can
    route; description instructs to create a dataplane first.
  - `address:string` (opt, `0.0.0.0`); `protocol:string` (opt, `HTTP`; enum `HTTP|HTTPS|TCP`);
  - `routeConfigName:string` (opt) ŌĆö binds route config; REQUIRED for traffic routing; must
    already exist.
  - `filterChains:array<object>` (opt) ŌĆö advanced full filter-chain config (TLS, tracing);
    use instead of `routeConfigName` for complex setups.
  *Out:* rich create. *Errors:* `ALREADY_EXISTS`, `CONFLICT` (port in use), `NOT_FOUND`
  (route config/dataplane).
- **cp_update_listener** ŌĆö Update listener. *Auth:* `listeners:update` (HIGH ŌĆö affects live
  traffic). *Params:* `name` (req, immutable identifier); `address`, `port` (>=1024),
  `protocol` (enum), `routeConfigName` (updates HCM route binding), `dataplaneId` (must exist
  & same team), `filterChains` (**REPLACES existing completely**) ŌĆö all opt. *Out:* update
  response.
- **cp_delete_listener** ŌĆö Delete listener. *Auth:* `listeners:delete` (HIGH). *Params:*
  `name` (req). *Out:* delete response.

### 3.3 Route configs / routes `[v1: src/mcp/tools/routes.rs]`

- **cp_list_route_configs** ŌĆö List route configurations. *Auth:* `routes:read` (SAFE).
  *Params:* `limit`, `offset`.
- **cp_get_route_config** ŌĆö Get route config by name (full virtual-host tree). *Auth:*
  `routes:read` (SAFE). *Params:* `name` (req).
- **cp_create_route_config** ŌĆö Create route config. *Auth:* `routes:create` (LOW). *Params:*
  `name:string` (req); `virtualHosts:array<object>` (req) ŌĆö each
  `{name, domains:[string], routes:[{name, match:{path|prefix|regex|template}, action:{cluster|weightedClusters|redirect}, ...}]}`.
  *Out:* rich create with `created: {virtual_hosts: N, routes: M}`.
- **cp_update_route_config** ŌĆö **Full replacement** of `virtualHosts`. *Auth:* `routes:update`
  (MEDIUM). *Params:* `name` (req), `virtualHosts:array<object>` (req).
- **cp_delete_route_config** ŌĆö Delete route config. *Auth:* `routes:delete` (HIGH). *Params:*
  `name` (req). *Errors:* `DEPENDENCY` if a listener binds it.
- **cp_list_routes** ŌĆö List all routes with metadata. *Auth:* `routes:read` (SAFE). *Params:*
  `limit`, `offset`, `routeConfig:string` (opt filter). (Direct-DB tool.)
- **cp_get_route** ŌĆö Get route by hierarchy (route_config ŌåÆ virtual_host ŌåÆ route). *Auth:*

## 4. Dynamic Gateway Tools (`api_*`)

`[v1: src/mcp/gateway/generator.rs, src/mcp/gateway/executor.rs, src/services/mcp_service.rs,
 src/mcp/handler.rs:1696-1889, src/mcp/tool_registry.rs:1077-1116]`

### 4.1 Source of truth & lifecycle

- Gateway tools are **rows in the `mcp_tools` DB table** with
  `category = GatewayApi`. Columns: `id, team (UUID), name, description, category,
  source_type (openapi|manual|learned|builtin), input_schema, output_schema,
  learned_schema_id, schema_source, route_id, http_method, http_path, cluster_name,
  listener_port, host_header, enabled, confidence, created_at, updated_at`.
- They are created/updated by `McpService` (REST endpoints, not MCP):
  `POST /api/v1/teams/{team}/routes/{route_id}/mcp/enable` (also bulk enable/disable,
  `PATCH .../mcp/tools/{name}`, `refresh-schema`). Enablement requires complete route
  metadata (`operation_id`, `http_method`, summary/description), which comes from an OpenAPI
  import, manual entry, or learned-schema enrichment; `confidence` carries the learned-schema
  confidence (1.0 for openapi/manual). Re-enabling refreshes `listener_port` and
  `host_header`. Learning aggregation can regenerate the tool with new schemas
  (`mcp_service.rs:616-665`).
- A tool's effective routing data: `listener_port` is resolved from the route config's bound
  listener; `host_header` is the first non-wildcard domain of the owning virtual host (None
  if only `*`); `gateway_host` is joined at execution time from the listener's dataplane.

### 4.2 Naming scheme

- With OpenAPI `operationId`: **`api_{operationId}`** (e.g. `api_getUser`).
- Without: **`api_{path_segments}_{method}_{routeId8}`** ŌĆö path split on `/`, `{param}` braces
  stripped, `-`ŌåÆ`_`, lowercased method (default `get`), plus the first 8 chars of the route
  UUID for uniqueness (e.g. `api_users_id_get_a1b2c3d4`).
- Description: metadata `summary`, else `description`, else `"{METHOD} {path}"`.

### 4.3 Input/output schema derivation

- Path parameters extracted from the route pattern via regex `\{([^}]+)\}`; each becomes a
  **required string property** `{"type":"string","description":"Path parameter: {name}"}`.
- If route metadata has `request_body_schema`, its `properties` and `required` are merged
  with the path-param schema (path params first; body-required appended without dupes).
- No params and no body ŌåÆ `{"type":"object","properties":{},"additionalProperties":false}`
  (MCP requires an object inputSchema). All generated schemas set
  `additionalProperties: false`.
- `output_schema` = metadata `response_schemas` verbatim (optional).

### 4.4 Invocation path (tools/call for `api_*`)

Authorization & execution order (`execute_gateway_tool`):

1. Static authz: any `api_*` name maps to `GATEWAY_AUTH = {resource:"api",
   action:"execute", risk: MEDIUM}` in the registry.
2. Filter session teams to those where `check_resource_access(ctx, "api", "execute", team)`
   passes; none ŌåÆ `Forbidden ("calling gateway tool requires api:execute scope")`.
3. Look up the tool by name across authorized teams (`get_by_name_with_gateway`, which joins
   route `exposure` and the listener's dataplane `gateway_host`); disabled tools are skipped;
   not found ŌåÆ `ToolNotFound`.
4. **Exposure re-check at execution time** (fp-nw8b.2): `tool.exposure` must be `"external"`,
   otherwise `Forbidden("Route is not externally exposed")` ŌĆö even with a stale grant.
5. GatewayTool agents additionally need a `GrantType::GatewayTool` grant whose `route_id`
   matches the tool's route (`Forbidden("Agent not granted access to this tool")`; tools with
   no route_id are rejected).
6. Category must be `GatewayApi` (else ToolNotFound "not an API tool").
7. `gateway_host` must be non-empty, else `Configuration` error instructing to create a
   dataplane and assign the listener to it.
8. **HTTP execution** (`GatewayExecutor`, shared `reqwest` client, **30 s timeout**):
   - URL: `http://{gateway_host}:{listener_port}{path-with-params-substituted}`. Path params
     substituted from arguments (string/number/bool accepted; missing param ŌåÆ
     `InvalidParams("Missing required path parameter: {p}")`).
   - `Host` header: `tool.host_header` if set, else `{gateway_host or 127.0.0.1}:{port}` ŌĆö
     needed for Envoy virtual-host matching.
   - Body = arguments minus path params. GET: body keys become **query parameters**;
     POST/PUT/PATCH: JSON body; DELETE: no body; other methods ŌåÆ
     `InvalidParams("Unsupported HTTP method")`.
   - Response ŌåÆ `ToolCallResult` with single text block
     `"HTTP {status_code} {reason}\n\n{body_text}"`; `isError: true` when non-2xx. Transport
     failure ŌåÆ `GatewayError` (-32603).

### 4.5 Team scoping & listing

- `tools/list` includes gateway tools per team (humans need `api:read` per team; GatewayTool
  agents see only granted route_ids; CpTool/ApiConsumer agents see none). Names are unique
  per team; lookup at call time scans the caller's authorized teams in order and takes the
  first enabled match.
- Note: `tools/list` does NOT filter by route exposure (only `enabled`); exposure is enforced
  at execution only ŌĆö see ┬¦7.

## 5. Resources & Prompts Surfaces

### 5.1 Resources ŌĆö YES, exposed `[v1: src/mcp/resources.rs]`

- URI scheme: **`flowplane://{type}/{team}/{name}`**, types `clusters | listeners | routes |
  filters`. MIME type `application/json` everywhere.
- `resources/list`: lists across the four types for the session's **first team** only.
  Requires an **org-bound session** (`admin:all` governance sessions without org context are
  Forbidden), org-scoped team resolution (cross-org team ŌåÆ generic
  `Forbidden("Team not accessible in your organization")`, deliberately not echoing the team
  string to prevent enumeration), and per-type `{type}:read` grant filtering (fp-xsli.2.4 ŌĆö
  previously any session could enumerate any team).
- `resources/read`: parses the URI (strict 3-segment format), same org binding + team
  resolution + `{resource_type}:read` grant check, then returns one `ResourceContent`
  `{uri, mimeType:"application/json", text: <pretty JSON>}`:
  - cluster: `{name, team, service_name, configuration, description, created_at, updated_at}`
  - listener: `{name, team, address, port, protocol, description, created_at, updated_at}`
  - route: queried from `route_configs` with columns `cluster_name, path_prefix, headers,
    route_order` ŌĆö **legacy column set, likely stale vs current schema** (see ┬¦7)
  - filter: `{name, filter_type, configuration, description, created_at, updated_at}`
- Unknown name ŌåÆ `ResourceNotFound` (-32601). Capabilities advertise `subscribe: false,
  listChanged: false` ŌĆö no subscriptions.

### 5.2 Prompts ŌĆö YES, exposed `[v1: src/mcp/prompts/]`

Static registry of **7 prompts**, all with arguments (every prompt requires `team`):

| Name | Description | Arguments |
|---|---|---|
| `debug_route` | Analyze route configuration issues and provide diagnostic insights | `route_name` (req), `team` (req) |
| `explain_filter` | Explain filter configuration and behavior in the request/response pipeline | `filter_name` (req), `team` (req) |
| `troubleshoot_cluster` | Diagnose cluster connectivity and health issues | `cluster_name` (req), `team` (req) |
| `analyze_listener` | Analyze listener configuration and provide optimization recommendations | `listener_name` (req), `team` (req) |
| `optimize_performance` | Suggest performance optimizations for Flowplane configuration | `team` (req) (+optional focus args) |
| `dev_deploy_and_verify` | Deploy a backend service and verify end-to-end connectivity through the gateway | `backend_host`, `backend_port`, `path`, `listen_port`, `team` |
| `dev_learn_and_document` | Observe API traffic and generate OpenAPI specification from learned schemas | `route_config_name`, `team` |

- `prompts/get` renders templated `PromptGetResult { description, messages:[{role:
  user|assistant, content:{type:text,...}}] }` via per-prompt renderer functions
  (`templates.rs`, ~1200 lines of workflow text). Missing required argument ŌåÆ
  `InvalidParams` naming the argument; unknown prompt ŌåÆ `InvalidParams("Prompt not found: ŌĆ”")`
  (note: NOT the dedicated `PromptNotFound` variant ŌĆö minor inconsistency).
- **No authorization is applied to prompts** (any authenticated session can list/render).

## 6. Session / Transport Details

`[v1: src/mcp/streamable_http/get_handler.rs, connection.rs, message_buffer.rs,
 notifications.rs, progress.rs, cancellation.rs, logging.rs]`

### 6.1 SSE stream (GET)

- Requires existing, format-valid session ID; registers a connection in the
  `ConnectionManager` with ID **`conn-{team}-{uuid-v4}`**, team taken as the first
  token-authorized team. **Per-team connection limit: 10**
  (`MAX_CONNECTIONS_PER_TEAM`); breach ŌåÆ HTTP 429 with message
  `Connection limit (10) exceeded for team: {team}`.
- The connection is attached to the session (`session.connection_id`); subsequent POSTs with
  `Accept: text/event-stream` deliver their JSON-RPC response on this stream as a `message`
  event and answer the POST with **202 Accepted** (headers only).
- SSE event types: `message` (JSON-RPC response), `progress` (progress notification), `log`
  (log notification), `ping` (heartbeat `{timestamp}` ms). Event data is the inner payload
  (not the enum wrapper). Keep-alive ping every **10 s** (`KeepAlive ... text("ping")`).
- Every event carries `id: {connection_id}:{sequence}` (monotonic per-connection `u64`).
- On client disconnect a `CleanupStream` Drop unregisters the connection and detaches it from
  the session. DELETE on the session also closes any attached connection.

### 6.2 Resumability & message buffering

- Each connection owns a `MessageBuffer`: ring buffer, **capacity 500 messages**, LRU
  eviction, atomic `u64` sequence counter (never reset; `clear()` keeps the counter).
- Reconnect: client sends `Last-Event-ID: {old_conn_id}:{seq}`. Server parses (last-colon
  split; invalid format ŌåÆ logged, start fresh), validates the old connection's **team equals
  the requesting team** (cross-team resume silently yields no replay ŌĆö no info leak), then
  replays buffered messages with `sequence > seq` in order, chained before the live stream.
  Expired/unknown old connection ŌåÆ fresh stream.

### 6.3 Notifications, progress, cancellation, logging

- Notification payload shapes (`notifications.rs`):
  - Progress: `{method:"notifications/progress", params:{progressToken, progress:f64,
    total?:f64, message?:string}}`.
  - Log: `{method:"notifications/message", params:{level, logger?, data}}`; levels are RFC
    5424: `debug < info < notice < warning < error < critical < alert < emergency`.
- `ProgressTracker` (`progress.rs`): `start(token, total?)` (duplicate token or non-finite/
  negative total ŌåÆ InvalidParams), `update(token, progress, message?)` enforcing **strict
  monotonic increase**, `complete(token)` removes. `ProgressToken` is an opaque string.
- `CancellationManager` (`cancellation.rs`): DashMap of `JsonRpcId ŌåÆ CancellationToken`
  (tokio_util); `register/cancel/complete/is_cancelled/child_token`.
- `McpLogger` (`logging.rs`): per-connection minimum log level filters; `should_log` returns
  true when no filter set or `level >= min`.
- **Wiring status in v1 (important for v2 scoping)**: `ProgressTracker`,
  `CancellationManager`, and `McpLogger` are implemented and re-exported but are NOT invoked
  from the live request path. The handler acknowledges `notifications/cancelled` and
  `logging/setLevel` with empty success without acting (the session has a `log_level` field
  the handler never updates). No tool emits progress notifications. v2 must decide: wire
  these up or drop the dead capability advertisement (`logging` capability is advertised).

## 7. Gaps and Smells (input to spec/08 & 08a)

Authorization / security:

1. **Origin validation not wired** ŌĆö `validate_origin_header` +
   `FLOWPLANE_MCP_ALLOWED_ORIGINS` exist (`security.rs`) and are documented as an MCP
   2025-11-25 requirement, but no streamable-HTTP handler calls them. CSRF defense-in-depth
   is absent in practice.
2. **Session ID as bearer-ish secondary credential** ŌĆö after `initialize`, any valid token
   presenting a known `MCP-Session-Id` can use that session (explicit comment in
   `post_handler.rs`). Authorization still derives from the presenting token, but session
   team binding is not re-checked; `SessionManager::validate_session_ownership` exists and is
   unused by the POST path.
3. **`tools/list` doesn't enforce route exposure for gateway tools** ŌĆö listing filters only
   on `enabled` (humans with `api:read` see tools for internal-exposure routes); exposure is
   enforced only at execution. The execution-path comment claims "tools/list only ever
   surfaces tools whose route is 'external'", which is not what the code does. Info-leak +
   confusing UX for agents.
4. **`Forbidden` maps to JSON-RPC -32600** (same code as a dozen other conditions); clients
   cannot reliably distinguish authz failures from malformed requests. Consider a custom
   code range in v2.
5. **`ops_*` authorization is borrowed** ŌĆö `ops_topology`, `ops_config_validate`,
   `ops_xds_delivery_status`, `ops_nack_history`, `devops_get_deployment_status`,
   `dev_preflight_check`, `cp_query_service` all gate on `clusters:read` even though they
   read listeners/routes/filters/xDS state; `ops_trace_request` gates on `routes:read`. A
   token with only `clusters:read` can see the full topology. v2 should define a real
   `ops:read`/`cp:read` resource (the registry description already hints "requires
   clusters:read **or cp:read**" but no `cp` resource exists).
6. **Registry contains 17 phantom authz entries** (certificates, WASM filters, reports,
   `cp_update_secret`, `cp_import_openapi`, `cp_start_learning`, `cp_list_schemas`,
   `cp_get_schema`, `cp_export_schema`, `cp_register/deregister_dataplane`) with no
   corresponding tool. `is_valid_cp_grant_pair()` derives the set of grantable
   (resource, action) pairs from this map, so phantom entries make phantom grants valid.
7. **Prompts and `ping`/`initialize` have no authorization**; prompts render team-named
   workflow text for any authenticated caller (low risk, but inconsistent).

Naming / schema consistency:

8. **Casing is mixed**: most tool params are camelCase (`routeConfig`, `targetSampleCount`)
   but rate-limit tools use snake_case (`requests_per_unit`, `org_name`, `team_name`) and
   ops tools mix (`dataplaneName` vs `resourceType`). Pick one convention in v2.
9. **`cp_stop_learning`** breaks the `verb_resource` pattern (cf.
   `cp_delete_learning_session`); `ops_learning_session_health` lives in `learning.rs` and
   dispatches via xds_state unlike its `ops_*` siblings.
10. **`team` argument inconsistency**: dataplane + rate-limit tools require explicit `team`;
    every other tool infers team from the session. Agents must learn two conventions.
11. **`cp_set_rate_limit_overrides` schema smell**: `max_*` fields are
    `["integer","null"]`, which the docs generator renders as type `any`; the doc-gen also
    truncates multi-line descriptions at the first line, producing dangling text ("per
    ┬¦14.1", "Returns 404 if", "`null` (or") in the published reference ŌĆö weak for LLM
    consumption.
12. **Output envelope inconsistency**: most tools use the `{ok/found, ref, data}` builders;
    rate-limit tools return raw response rows; gateway tools return raw
    `"HTTP {code}\n\n{body}"` text; ops tools return bespoke JSON. v2 should standardize
    (ideally with `outputSchema` declared ŌĆö only gateway tools populate `output_schema`
    today).
13. **Tool descriptions vary wildly in quality**: filters/rate-limit have rich multi-line
    LLM-oriented descriptions with examples; `cp_create_rate_limit_domain`'s schema relies on
    the description for constraints; several (`cp_query_port`, `cp_list_clusters`) are
    one-liners. Risk prefix `[Risk: X]` is prepended at list time rather than being a
    structured annotation (MCP `annotations.hints` exists in the protocol types but is unused).

Protocol / behavior:

14. **Dead capabilities**: logging is advertised but `logging/setLevel` is a no-op;
    cancellation notifications are acknowledged but nothing is cancelled; progress
    infrastructure unused (┬¦6.3). Resources advertise `subscribe:false` correctly but the
    resources `routes` reader queries legacy columns (`route_configs.cluster_name`,
    `path_prefix`, `route_order`) that don't match the current normalized route schema ŌĆö
    likely a runtime SQL error on read (`resources.rs:425-435`).
15. **`tools/mod.rs::execute_tool`** is a vestigial second dispatcher (handles only 3 tools;
    real dispatch lives in handler.rs) ŌĆö two sources of truth for routing.
16. **Pagination defaults inconsistent**: lists default 50/max 1000, but `ops_audit_query`
    is 20/100 and `ops_nack_history` 10/100.
17. **GET-arg flattening on gateway tools**: for GET tools, body-schema fields silently
    become query params; non-string values are serialized via `to_string()` (JSON syntax for
    objects/arrays) ŌĆö undocumented and surprising.
18. **`initialize` does not enforce session ordering** (any method works without initialize);
    acceptable for stateless HTTP but should be an explicit v2 decision.
19. **SSE registration uses only the first authorized team**; multi-team tokens cannot
    receive notifications scoped to their other teams, and the 10-connection limit is
    per-team-name (a noisy team can starve a shared dashboard).
